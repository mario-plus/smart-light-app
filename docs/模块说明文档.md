# SmartApp 模块说明文档

## 1. 项目定位

`SmartApp` 是一个面向智慧城市/智慧园区场景的 Android 运维管理应用，核心目标是把多类物联网设备接入到统一的移动端管理入口中，实现：

- 设备总览与运行态监控
- 设备列表、详情与历史数据查询
- 站点与地图点位管理
- 智慧路灯业务管理与控制
- 智慧屏业务管理
- 环境感知设备数据查看
- 摄像头实时监控
- 用户、项目、系统配置管理

从当前代码实现来看，它更偏“运维控制台”而不是普通用户 App。

---

## 2. 技术栈与实现方式

- 开发语言：Kotlin
- UI：Jetpack Compose + Material 3
- 页面导航：Navigation Compose
- 网络层：Retrofit + OkHttp + Gson
- 分页：Paging 3
- 图片加载：Coil
- 地图：高德地图 SDK
- 视频与流媒体：
  - Media3 / ExoPlayer 体系
  - WebRTC 用于摄像头实时视频
- 本地存储：
  - SharedPreferences 用于 token、当前项目等轻量数据
  - DataStore 用于模块开关与系统配置持久化

## 3. 整体架构逻辑

## 3.1 架构分层

项目整体采用的是“Compose 页面 + ViewModel 状态管理 + Retrofit Service 直连接口”的轻量分层结构，没有单独再抽 Repository 层。

### 1）入口与导航层

- `MainActivity` 负责：
  - 初始化地图隐私协议
  - 初始化 `RetrofitClient`
  - 启动 Compose 根页面
- 根页面 `SmartStreetLightApp` 负责：
  - 登录态切换
  - 底部导航
  - 页面路由跳转
  - 在复杂页面间缓存部分 ViewModel

### 2）UI 页面层

- 目录位置：`app/src/main/java/com/unilumin/smartapp/ui/screens`
- 职责：
  - 页面布局与交互展示
  - 订阅 ViewModel 的 `StateFlow`
  - 响应点击、筛选、切换、跳转等行为

### 3）ViewModel 业务层

- 目录位置：`ui/viewModel`
- 职责：
  - 维护页面状态
  - 拼装接口参数
  - 触发分页加载、详情查询、控制命令
  - 将原始接口结果转换成 UI 可直接消费的数据结构

当前项目里，ViewModel 是业务核心层，承担了较多数据整形和状态编排职责。

### 4）接口服务层

- 目录位置：`client/service`
- 按业务域拆分 Retrofit Service：
  - `AuthService`
  - `DeviceService`
  - `RoadService`
  - `ScreenService`
  - `SiteService`
  - `ProjectService`
  - `UserService`
  - `SystemService`

### 5）配置与基础能力层

- `RetrofitClient`：统一构建 OkHttp + Retrofit
- `UniCallbackService`：统一解析两类后端响应结构
- `AuthInterceptor`：自动注入 Bearer Token
- `SystemConfigManager`：保存设备类型/业务模块开关配置
- `ServerConfig`：维护当前服务端地址

## 3.2 运行流程

### 1）应用启动

1. `MainActivity` 初始化地图 SDK 与网络客户端
2. 按 `ServerConfig.getBaseUrl()` 构建 Retrofit
3. 进入 Compose 根页面

### 2）登录流程

1. 用户输入账号密码
2. 先请求公钥
3. 使用 RSA 加密密码
4. 调用登录接口
5. Token 写入本地
6. 进入主界面

### 3）主界面运行逻辑

1. 底部四大主页签驱动整体功能入口
2. 各页面创建自己的 ViewModel
3. ViewModel 内部通过 `StateFlow + PagingFlow` 组织数据
4. 页面筛选条件变化时，自动触发分页流重新加载

### 4）退出登录

1. 清空 token
2. 重置登录态
3. 重新回到登录页

---

## 3.3 状态管理逻辑

项目主要使用 `MutableStateFlow` + `collectAsState()` 组织状态。

常见模式：

- 普通页面状态：
  - 搜索词
  - 筛选状态
  - 当前选中对象
  - loading 状态
- 分页状态：
  - 通过 `BaseViewModel.createPagingFlow()` 创建
  - 使用 `combine(...)` 将多个筛选条件合并成参数流
  - 条件变化后自动重建 `Pager`

这使得设备、路灯、屏幕、站点等模块可以复用同一套分页逻辑。

---

## 3.4 基础抽象：BaseViewModel

`BaseViewModel` 是整个项目的公共基类，主要提供：

- `isLoading` 公共加载状态
- `totalCount` 总数状态
- `launchWithLoading()`：
  - 统一 loading 控制
  - 统一异常捕获
  - 统一 Toast 提示
- `launchDirect()`：
  - 适合轻量请求或不需要 loading 的场景
- `createPagingFlow()`：
  - 统一封装 Paging 3
- `fetchPageData()`：
  - 统一分页接口解析

这说明项目在“业务页面很多”的前提下，已经做了基础复用，降低了重复代码。

---

## 3.5 接口与异常处理逻辑

项目后端存在两套响应结构：

- `ResponseData<T>`
- `NewResponseData<T>`

`UniCallbackService` 统一负责：

- Retrofit `Call` 转挂起函数
- HTTP 层失败处理
- 业务码校验
- 抛出 `ApiException`

这套设计让 ViewModel 可以直接写成同步风格的协程代码。

---

## 3.6 鉴权与请求拦截逻辑

### Token 管理

- 登录成功后 token 保存在 `SharedPreferences`
- `TokenManagerFactory` 负责返回 token 管理实例
- `AuthInterceptor` 在每次请求中自动写入：
  - `Authorization: Bearer xxx`
  - `Accept-Language: zh`

### 特殊放行

以下请求不带 token：

- 请求头显式带 `No-Auth`
- MinIO 文件地址请求

---

## 3.7 本地配置逻辑

项目有两类本地配置：

### 1）运行环境配置

- 协议、IP、端口由 `ServerConfig` 维护
- 登录页可动态修改服务端地址
- 保存后会重新初始化 Retrofit

### 2）功能开关配置

由 `SystemConfigManager + DataStore` 维护：

- 设备列表展示哪些设备类型
- 设备页展示哪些智慧应用
- 智慧路灯页面展示哪些子模块
- 智慧屏页面展示哪些子模块
- 智慧感知展示哪些产品类型

这意味着项目具备“按项目/交付场景裁剪功能入口”的能力。

## 5.2 分页体系

分页通过以下链路实现：

1. ViewModel 定义筛选参数 `StateFlow`
2. `combine(...)` 合并为参数流
3. `BaseViewModel.createPagingFlow()` 构建 `Pager`
4. `GenericPagingSource` 执行实际请求
5. 页面通过 `collectAsLazyPagingItems()` 消费

该设计在设备、站点、路灯、智慧屏等模块中被广泛复用。

## 5.3 配置裁剪体系

- 设备页展示哪些应用入口

- 智慧屏展示哪些功能

这是一套“前端可裁剪能力配置”机制，适合项目制交付。

## 6. 业务接口域划分

接口路径常量集中在 `RequestPathKey` 中，后端按域划分为：

- `SMART_COMMON_APPLICATION`
  - 通用基础能力
  - 路段、站点、环境、摄像头地址、产品规则
- `SMART_ROAD_LAMPS`
  - 路灯控制、分组、策略、任务、统计
- `SMART_LED`
  - 智慧屏、节目、分组、方案、素材
- `IOT_HUB_MANAGER`
  - 设备分页、详情、实时数据、历史数据、离线分析
- `CAS`
  - 登录、公钥、用户信息、项目切换、系统信息
- `ZLM`
  - WebRTC 信令交换

这种划分使服务边界比较清晰。

## 7. 当前架构特点总结

## 优点

- 页面与业务域划分清晰
- ViewModel 复用度高
- 分页抽象统一
- 配置裁剪能力完整
- 路灯业务能力较丰富
- 视频、地图、图表等能力接入完整

## 当前不足

- 缺少 Repository 层，ViewModel 直接调用 Service，业务耦合偏高
- 站点详情逻辑尚未闭环
- 智慧广播模块尚未实现
- 监控云台控制仍为占位
- `RetrofitClient` 当前对 SSL 采取全信任策略，适合开发联调，但生产环境有安全风险

## 9. 结论

`SmartApp` 当前已经形成了一个以“设备运维 + 智慧路灯”为核心、并逐步扩展到站点、智慧屏、感知、监控等模块的移动端运维平台。

从代码成熟度看：

- 已较成熟：登录、设备中心、概览、智慧路灯、智慧屏、个人中心
- 基本可用：智慧感知、监控视频
- 待补完：站点详情、智慧广播、监控 PTZ 控制

整体上，这是一个围绕智慧城市物联网设备管理构建的 Android 原生运维应用，具备较明确的模块边界和可扩展性。
